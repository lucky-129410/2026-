从底层网络协议栈到应用层发生的完整过程。

### 1. DNS 解析 (Domain Name System)
在发送请求之前，程序只知道域名（`movie.douban.com`），但网络层通信需要目标服务器的 **IP 地址**。
* **查找顺序：** 1.  **本地缓存：** 首先检查程序自身的 DNS 缓存、操作系统的缓存。
2.  **Hosts 文件：** 检查本地操作系统的 `hosts` 文件是否有手动配置。
3.  **递归/迭代查询：** 如果本地没有，会向本地 DNS 服务器（通常由 ISP 提供）发起请求。本地 DNS 服务器会依次询问根域名服务器、顶级域名服务器（.com）、权威域名服务器（douban.com），最终返回该域名的 IP 地址。

### 2. 建立 TCP 连接 (Transmission Control Protocol)

有了 IP 地址后，客户端需要与服务器建立一个可靠的传输通道。因为是 HTTPS，默认连接服务器的 **443 端口**。

* **三次握手过程：**
1. **SYN：** 客户端发送一个带 SYN 标志的数据包给服务器，表示想建立连接。
2. **SYN + ACK：** 服务器收到后，回复一个带 SYN 和 ACK 标志的数据包，表示同意连接并确认收到。
3. **ACK：** 客户端再次发送一个 ACK 包，确认收到服务器的同意。


* **目的：** 确保双方都有发送和接收数据的能力。

### 3. TLS/SSL 握手 (Transport Layer Security)

这是 HTTPS 与 HTTP 的核心区别。在 TCP 连接建立后，需要进行加密协议的握手，以确保数据传输的安全（防窃听、防篡改）。

* **主要流程：**
1. **Client Hello：** 客户端告知自己支持的加密算法版本。
2. **Server Hello + 证书：** 服务器选定加密算法，并下发自身的 **数字证书**（包含公钥）。
3. **验证证书：** 客户端验证证书是否合法（由受信任的机构签发）。
4. **生成会话密钥：** 客户端生成一个随机的“对称密钥”，用服务器的公钥加密后传给服务器。
5. **开始加密通信：** 之后的所有数据（包括 HTTP 请求体）都使用这个对称密钥进行加解密。



### 4. 发送 HTTP 请求

加密通道建好后，`Requests` 库开始构建真正的 HTTP 报文。

* **报文结构：**
* **请求行：** `GET / HTTP/1.1`
* **请求头 (Headers)：** 包含 `Host: movie.douban.com`、`User-Agent`（爬虫身份标识）、`Accept` 等。
* **空行**
* **请求体：** 对于 GET 请求，通常为空。


* **传输：** 这些数据被切分成一个个 TCP 分段，通过底层链路发送给服务器。

### 5. 服务器处理并响应

服务器接收到加密的 TCP 包，解密出 HTTP 请求。

* **处理流程：**
1. **反向代理 (如 Nginx)：** 接收请求，可能负责负载均衡，并将请求转发给后端的应用服务器（如 Python 的 Gunicorn/Django/Flask）。
2. **业务逻辑：** 后端程序查询数据库（豆瓣电影数据），渲染 HTML 页面或生成 JSON 数据。
3. **返回响应：** 封装成 HTTP 响应报文（包含状态码 200 OK，响应头，以及电影网页的内容）。



### 6. TCP 四次挥手 (Connection Termination)

数据交换完成后，需要断开连接以释放资源。

* **过程：** 双方互发 FIN (Finish) 和 ACK (Acknowledgment) 包。
* **注意点：** * 如果在请求头中包含了 `Connection: keep-alive`，则 TCP 连接会保持一段时间不销毁，以便后续请求复用，从而减少重复握手的开销。

### 面试官考察重点总结：

1. **全栈理解：** 你是否理解从应用层 (HTTP) -> 安全层 (TLS) -> 传输层 (TCP) -> 网络层 (IP/DNS) 的每一层职责。
2. **HTTPS 原理：** 特别是 TLS 握手过程中非对称加密（交换密钥）和对称加密（传输数据）的结合。
3. **工程细节：** 比如 `User-Agent` 的作用（爬虫识别）、`Keep-Alive` 的性能优化等。
